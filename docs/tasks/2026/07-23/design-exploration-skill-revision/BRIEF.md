# 설계 탐색 스킬 개선

## Goal

- Project Legibility가 첫 해법이나 선호 구조를 정답 형태로 강요하지 않고, 목적에 맞는 탐색과 검증 가능한 구조 판단을 돕도록 핵심 스킬을 조정한다.

## Scope

- `purpose-fit-design`과 `structure-first`의 역할·호출·실행 문구를 필요한 범위에서만 바꾼다. 제품 계약은 달라진 호출 경계만 반영하고 기존 제품 약속과 구성 중심을 유지한다.
- 전문 스킬, 새 통합 스킬, 자동 판정기, 점수 기반 중단 조건과 배포를 막는 행동 검증 자동화는 이번 작업에서 제외한다.
- Project Legibility 전체의 정체성이나 모든 스킬에 적용할 판단 철학을 다시 정의하지 않는다.

## Current Understanding

- 두 정본 스킬 안에서 이상 징후와 경험칙은 조사 단서로 사용하며, 그 자체를 구조 변경이나 범위 확대의 근거로 삼지 않는다.
- `purpose-fit-design`은 새 기능, 기존 구현 재사용과 임시 구현의 초기 방향을 사용자·도메인 목적, 제약, 근거와 성공 조건에 맞춰 정해야 한다. 대안 비교는 중요한 선택에만 사용하고, 탐색만으로 구현 범위를 넓히지 않는다.
- `structure-first`는 기능 구현·버그 수정·리팩터링 전반에서 읽기 쉬운 흐름과 책임, 조합, 부작용·상태 경계와 계약 검증을 다루는 기본 관점을 유지해야 한다. `Primary Flow`, `Atom`, 조합 지점과 부작용 경계는 필수 결과물이 아니라 필요한 만큼 선택하는 수단이어야 한다.
- 제품 약속과 핵심 구성은 유지하며, 달라진 참여 조건과 경계는 해당 스킬의 제품 역할에만 반영한다.
- 기준안과 후보안을 짝지은 비교 시나리오에서 두 스킬의 판단 차이를 관찰한다.

## Current State

- 제품 계약은 기존 약속과 구성 중심을 보존하면서 두 스킬의 달라진 호출 경계만 남기도록 정리했다. README와 플러그인 명세의 제품 중심 문구도 기존 방향을 유지하고 두 스킬 설명만 갱신했다.
- `purpose-fit-design`을 이번 작업의 “설계 공간 재개방” 문제에 치우치지 않는 일반적인 초기 방향 판단 스킬로 정리했고, `structure-first`를 구체적인 구조 문제가 이미 드러난 경우로만 좁혔던 중간안도 폐기했다.
- `purpose-fit-design`과 `structure-first` 정본 변경을 검증해 원격 저장소에 올리고 플러그인 잠금 정보와 생성 스냅샷을 갱신했다.
- 기준안과 후보안을 가린 세 블라인드 비교에서 후보안이 전체적으로 우세했고 명백한 실패는 없었다.
