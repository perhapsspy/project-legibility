# 스킬: Structure First

> 영문 기본 문서: `SKILL.md`
> 설명 동기화: 변경이 여러 단계의 흐름, 상태 생명주기, 부작용·판단 소유권, 단위 간 조합이나 경계 계약을 새로 만들거나 재편하는 코드 생성, 기능 작업, 버그 수정, 리팩터링과 코드 리뷰에 씁니다. 기존 구조 때문에 이런 요소를 추적하거나 검증하기 어려운 경우에도 씁니다. 구조 변경은 선택 사항입니다. 기계적 수정, 일회성 실험과 일관된 국소 변경은 현재 책임 단위와 집중된 검증으로 처리합니다.

## 목적

> **주 흐름(Primary Flow)**: 명령형으로 흐름을 조정하는 것이 자연스러운 로직을 위에서 아래로 읽을 수 있게 드러낸 주 경로

관련 동작이 그 코드에 자연스러운 형식으로 읽히게 합니다. 요청된 동작을 이해·변경·검증할 수 있을 만큼 책임, 조합, 부작용, 상태와 검증을 분명하게 만듭니다. 주 흐름, 독립 역할(Atom), 조합 지점과 부작용 경계는 관련 동작을 더 분명하게 만들 때 사용합니다.

복잡도를 없애거나 실제로 독립적인 책임 안에 가두는 변경을 선호합니다. 기존 구조가 일관된 변경을 이미 수용하면 그대로 두고, 바뀐 동작은 구현 세부를 따라가는 테스트가 아니라 **계약 중심 테스트**로 보호합니다.

## 적용 범위

- 흐름, 상태 생명주기, 부작용·판단 소유권, 단위 간 조합이나 경계 계약을 새로 만들거나 재편하는 코드 생성, 기능 작업, 버그 수정, 리팩터링과 코드 리뷰에 쓴다. 이들을 추적하거나 검증하기 어려운 기존 문제에도 쓴다.
- 현재 구조가 이미 분명하면 가벼운 관점으로만 적용한다.
- 계획 수립, 분류, 범위 분석에서는 응답 형식으로 강제하지 말고 내부 판단 관점으로만 쓴다.
- 사용자에게 보이는 버그나 UI 오작동은 먼저 관찰 가능한 동작을 규명하고, 그 동작에 대한 현재 작업 단위의 책임이 분명해진 뒤에만 구조 작업을 적용한다.
- 기계적 수정, 일회성 실험과 일관된 국소 변경은 현재 책임 단위와 집중된 검증으로 처리한다.

## 핵심 원칙

- 추적 가능한 동작과 소유권 > 구조적 단순성 > 재사용성 > 추상화
- 미래 가능성보다 **현재 코드의 명확성**을 우선한다.
- 그대로 둠, 인라인화, 합침, 삭제, 순서 변경과 추출은 모두 정상적인 결과다.

## 적용 절차

증상이나 결과가 나타난 위치만 따르지 말고, 바꾸려는 동작이나 규칙을 책임지는 가장 작은 **현재 작업 단위**를 고른다.
국소 수정은 보통 함수나 파일, 기능 작업은 모듈이나 유스케이스, 더 큰 리팩터링은 기능 묶음이나 하위 체계가 현재 작업 단위가 된다.

1. **동작 추적**
- 의도와 가장 작은 관찰 가능한 완료 조건을 한 문장으로 적는다.
- 명령형 흐름, 상태 전이, 이벤트 생명주기, 규칙 집합, 데이터 흐름, 프로토콜 경계처럼 그 코드에 자연스러운 읽기 형식으로 동작을 따른다.
- 그대로 두면 요청 동작, 계약이나 의미 있는 검증이 깨지는 호출부, 타입, 테스트, 문서, 설정만 포함하고 인접 정리는 제외한다.

2. **구조 요구나 마찰 확인**
- 변경이 새로 만들거나 재편하는 구조 요구를 확인한다. 흐름, 상태 전이·생명주기, 부작용·완료 소유권, 조합이나 경계 계약이 해당할 수 있다.
- 기존 동작을 추적·변경·검증하기 어렵게 하는 문제도 확인한다.
- 현재 단위가 변경과 집중된 검증을 이미 맡고 있다면 작업을 그 단위 안에서 끝낸다.

3. **의미 있는 구조 선택만 탐색**
- 국소적인 명확화와 구조 변경이 둘 다 신뢰할 만한 후보일 때만 비교한다.
- 그대로 둠, 인라인화, 합침, 삭제, 순서 변경과 추출을 동등한 후보로 다룬다.
- 주 흐름은 명령형 흐름 조정을 명확하게 할 수 있다. **독립 역할(Atom)**은 관련 없는 책임과 조율하지 않고도 동작을 바꿀 수 있으며 그 자체로 이해 가능한 역할이다. 크기, 순수성, 추출 여부는 이러한 독립성을 뒷받침할 수 있지만 필수 조건은 아니다.
- 조합 지점은 자연스러운 소유자에게서 찾을 수 있게 하고, 서로 다른 판단과 생명주기는 각자의 소유자에 둔다.
- 부작용의 소유권과 실패 의미가 보이게 한다. 별도 경계가 추론, 검증, 재시도나 교체를 실질적으로 개선할 때만 부작용을 격리한다.

4. **전체 효과로 선택**
- 복잡도를 없애거나 독립 책임 안에 가두는 구조 변경을 선택한다.
- 도우미 함수 연쇄, 감싸기 계층, 문맥 객체, 설정, 상태, 오류 채널과 생명주기를 포함한 전체 경로에서 복잡도가 어디에 놓였는지 확인한다.
- 미확정 선택에 사용자 권한이 필요하거나 공개 API, 데이터, 보안, 의존성·비용, 마이그레이션에 되돌리기 어려운 약속을 만들 때 결정을 요청한다. 그 밖의 선택에는 되돌릴 수 있는 국소 가정을 사용하고 알린다.

## 확장과 소유권

- 공개 I/O/시그니처는 확정된 책임만 담는 최소 형태로 시작하고, 책임이 실제로 바뀔 때만 키운다(새 외부 입력, 의미 혼합, 경계 이동).
- 확인되지 않은 미래 사용을 위해 옵션, 설정, 의존성, 추상화를 추가하지 않고, 이번 변경으로 쓰이지 않게 된 코드는 제거한다.
- 정책, 판단, 계산 규칙마다 서로 경쟁하지 않는 결정 경로를 둔다. 단일 소유자, 명시적인 조합 순서, 조정·충돌 해결 규약 중 하나일 수 있다.
- 외부에서 관찰되는 쓰기의 소유자를 찾을 수 있게 한다. 여러 쓰기 주체가 의도된 경우에는 억지로 중앙화하지 말고 조정 방식이나 충돌 처리 계약을 드러낸다.
- 더 큰 범위에서는 관련 진입점, 조합 또는 조정 방식, 판단이 결정되는 경로를 찾을 수 있게 한다.
- 비동기 경계에서는 어떤 결과를 최신으로 보고 언제 완료로 판정하는지 결정 경로를 명시한다. 단일 소유자일 수도, 규약일 수도 있다.
- 판단 규칙의 소유가 바뀌거나 새 동등 경로를 넣는다면, 가능하면 같은 변경에서 기존 경로를 제거하거나 비활성화한다. 어렵다면 소유자와 종료 조건을 포함한 단계적 전환 계획을 함께 남긴다.
- `판단 규칙`: 동작을 결정하는 반복 조건, 가중치, 우선순위, 정책, 계산, 키 생성 규칙.
- `동등 경로`: 외부에서 관찰되는 결과가 같은 대체 실행 경로.

## 테스트

- 가능한 한 책임을 소유하는 가장 안정적인 단위에서 테스트를 충분히 작성
- 구현 세부가 아니라 현재 작업 단위와 소유 역할/경계 사이의 **계약(I/O, 불변조건, 경계값)**을 검증
- 위험과 변경 종류에 검증을 맞춘다. 버그는 수정 전에 재현하거나 현재 동작을 규명하고, 리팩터링은 전후의 안정 동작을 확인하며, 기능은 성공·실패·관련 경계를 검증한다. 가능한 원인을 여러 개 한꺼번에 바꾸기 전에 실패를 좁히고, 검사를 실행할 수 없으면 이유와 다음으로 유용한 검사를 밝힌다.
- 흐름 조정이나 경계 통합이 주요 위험이라면 현재 작업 단위를 직접 테스트한다
- 비동기 또는 상태 경계가 있다면 최신이 아닌 결과의 처리, 완료 처리의 균형, 동일 입력을 다시 받아도 관찰 가능한 상태나 효과가 바뀌지 않음 같은 소유권 계약을 그 경계를 소유한 가장 안정적인 단위에서 검증한다
- 이번 변경에서 테스트를 추가할 수 없다면 그 사실을 명시하고, 다음 안정적인 책임 단위와 필요한 계약 케이스를 함께 적는다
- 테스트 코드도 읽히게 유지한다. `each`나 표 기반 사례로 중복을 줄이고, 구조를 흐리지 않는 작은 도우미만 허용하며, 테스트 하나는 핵심 주장 하나에 집중한다.

## 피해야 할 패턴

- 분리 후 인자/상태 전달이 늘어나면 되돌리기
- 보기 좋으려고만 함수/파일 분리하지 않기(유틸 난립 금지)
- 상위 흐름을 짧게 보이려고 복잡도를 도우미 함수 연쇄, 감싸기 계층, 문맥 객체, 설정, 숨은 상태, 오류 채널이나 추가 생명주기로 옮기지 않기
- 이름이 설명문처럼 길어지면 경계를 재검토
- 미래 재사용 가정의 추상화/레이어 추가 지양
- 테스트에서 과도한 추상화나 도우미 난립 지양
- "나중에 필요할 수도 있음" 이유로 인자 선추가 금지
- 동일 정책, 판단, 계산, 키 생성 규칙의 다중 소유 금지
- 여러 쓰기 주체를 조정하는 일 자체가 명시적 책임이 아닌데도 같은 외부 관찰 결과를 여러 위치에서 쓰지 않기
- 소유권과 초기화 의미가 분명하지 않은데도 상위 경계의 상태를 국소 가변 상태로 동기화해 복제하지 않기
- 한 단위가 읽은 입력이나 갱신 경로에 다시 써서 자기 되먹임 고리를 만들지 않기
- 소유자와 종료 조건을 갖춘 단계적 전환 계획 없이 같은 결과를 내는 새 경로와 기존 경로를 함께 유지하지 않기

## 위험에 맞춘 검토

현재 변경과 관련 있는 질문만 사용한다.

- 관련 동작과 판단 소유권을 전보다 쉽게 추적할 수 있는가?
- 복잡도가 제거되거나 독립 책임 안에 갇혔는가, 아니면 도우미, 감싸기 계층, 문맥 객체, 상태, 오류 채널이나 생명주기로 이동했는가?
- 새 경계가 실제 책임, 부작용이나 계약에 대응하는가?
- 바뀐 관찰 동작과 계약을 검증이 보호하는가?

## 선택적 완료 근거

이 템플릿은 구조 자체를 설명하는 것이 핵심일 때만 쓴다. 그 외에는 자연스럽게 응답한다.

형식이 실제로 유용한 경우에만 아래 항목을 제시:

- `Current Unit:` function/file | module/use case | capability/subsystem
- `Structural Demand:` 달라진 흐름, 상태, 부작용, 소유권, 조합이나 계약, 기존 구조 문제 또는 `none; structure unchanged`
- `Readable Behavior:` 변경 뒤의 자연스러운 인과/읽기 형식
- `Structural Choice:` `kept` | `inlined` | `merged` | `deleted` | `reordered` | `extracted`와 그 이유
- `Tests:` `added ...` 또는 `deferred because ...; next stable unit(s): ...; required contract cases: ...` (해당되면 freshness/completion contract 포함)

판단 규칙 소유가 바뀌는 리팩터링에서는 아래 항목을 함께 제시:

- `Decision Ownership:` `policy/decision/calculation/key-generation rule -> owner unit`; 중복 소유 제거 여부 yes/no

시그니처/경계가 커졌거나 구경로를 대체한 리팩터링에서는 아래 1줄을 함께 제시:

- `Refactor Check:` 인자 증가 근거 / 구경로 상태(제거·비활성화·마이그레이션 계획)
